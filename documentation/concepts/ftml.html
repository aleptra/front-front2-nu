FTML (Front Text Markup Language) is the language of Front. It extends standard HTML with FTML - a declarative layer that allows you to define structure, style, layout, interactivity, and behavior directly through attributes, without writing any JavaScript or CSS.

Fundamentals

FTML is built on a small set of fundamentals that define how interfaces are described and composed: Attributes, Variables, Pages, Templates, Modules, and Plugins. These elements provide structure, state, reuse, and extensibility directly in markup, forming a cohesive model for building declarative and reactive interfaces.

Low-Code

FTML embodies a Low-Code approach by allowing developers and designers to define application structure, styling, behavior, and interactivity directly through markup attributes. Common patterns like loops, conditionals, computed values, event handling, and reusable templates are all expressed declaratively without writing JavaScript or CSS, reducing boilerplate and accelerating prototyping.

This attribute-driven, declarative syntax enables rapid development, consistent application behavior, and maintainable code while allowing teams to collaborate seamlessly within a single markup language.

FTML vs Traditional Markup

FTML simplifies interface development by overcoming limitations of traditional markup languages like HTML, XHTML, and XML, which often lack dynamic capabilities and require external scripts. Structure, logic, styling, interactivity, templates, and reactive updates are all defined declaratively in markup. The table below highlights these differences.

Comparison of FTML with Traditional Markup:
Feature Traditional Markup FTML
Structure ❌ Static ✅ Static + dynamic
Interpretation ❌ Browser renders static markup ✅ Browser renders compiled output
Logic ❌ Requires external scripts ✅ Declarative logic in markup
Styling ❌ Separate CSS stylesheets ✅ Declarative styling in markup
Dynamic content ❌ Requires scripts and templating ✅ Bindings & loops in markup
Interactivity ❌ JavaScript event handlers ✅ Declarative event handling
Components ⚠️ Manual reuse via includes or scripts ✅ Reusable components
Reactive updates ❌ Manual DOM manipulation ✅ Auto DOM sync
Templates ⚠️ Separate template engines ✅ Reusable templates in markup
Computed values ❌ Computed in scripts ✅ Computed in markup
Data-driven rendering ❌ Requires scripting ✅ Reactive rendering
Conditionals ❌ Scripted conditionals ✅ Conditional rendering in markup
Event handling ⚠️ Inline or external JavaScript ✅ Inline declarative events
