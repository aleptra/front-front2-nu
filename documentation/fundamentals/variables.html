Variables in Front allow you to store, manipulate, and bind data dynamically in your FTML markup. They can exist globally, locally within elements, or be provided by modules and plugins.

  • Global Variables: Accessible from anywhere in your application using bindglobal.
  • Element Variables: Scoped to a particular element or template using bindvar.
  • Mapped Variables: Variables that can be dynamically computed or transformed via mapbindvar.

Variables are reactive: changes propagate automatically to elements bound to them, eliminating the need for manual DOM updates.

Variable Binding

Bind variables to elements using attributes in FTML. Common binding attributes include:

  • bindvar: Bind a local variable to an element's content or attributes.
  • bindglobal: Bind a global variable to any element in your application.
  • bindasset: Bind to structured data cached or loaded from external JSON or assets.
  • bindfield: Bind variables to input fields, select boxes, or other form elements for dynamic updates.

Variables support multiple bindings simultaneously and can interact with plugins or modules to create dynamic, data-driven UIs.

Scope and Reactivity

Understanding scope is critical for variable management:

  • Global Scope: Variables accessible everywhere. Ideal for application-wide settings, shared data, or theme values.
  • Local Scope: Variables restricted to a template or element. Useful for loops, forms, or reusable components.
  • Dynamic Mapping: Use mapbindvar to transform or compute variables on the fly based on other variables or external data.

All bindings are reactive: changing a variable updates all associated elements automatically.

Best Practices

  • Name variables descriptively to indicate their purpose and scope.
  • Prefer bindglobal only for data that truly needs to be shared across templates.
  • Use bindvar for local or reusable templates to avoid global namespace collisions.
  • Keep dynamic mappings simple and readable; complex computations are easier to manage in plugin methods.
  • Always initialize variables before binding to prevent runtime errors.
